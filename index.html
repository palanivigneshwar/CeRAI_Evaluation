01 — System Under Test

What was evaluated and why

The target system is Gemma3n:e2b, Google's compact multilingual model, served locally via Ollama. It was chosen for three reasons: it supports Indian languages natively, it runs on consumer hardware (RTX 3050 4GB), and it represents the class of models most likely to be deployed in low-resource, offline-first settings relevant to the Gates Foundation's India programmes.

The evaluation used the CeRAI AIEvaluationTool's Conversational_Quality test plan with the Dialogue_Coherence metric — assessing whether the model maintains context across multi-turn conversations and produces coherent, contextually appropriate responses.

02 — Test Suite Design

Test suite and reasoning

The CeRAI tool ships with 670 test cases. Rather than building custom test cases, this evaluation used the existing DataPoints.json corpus filtered to the agriculture domain — the most relevant domain for the fellowship's focus on food security and rural development — with language left unfiltered to capture multilingual behaviour naturally.

The Dialogue_Coherence metric was chosen because it tests the property most critical for real-world deployment: whether a chatbot maintains context across a conversation, not just whether it answers individual questions correctly. For a farmer asking about crop disease mid-conversation about irrigation, context persistence is non-negotiable.

03 — Results

What the results show

The model completed all 10 test cases. The response analyzer returned real scores for 3 cases and 0 for 7. The 3 scoring cases used language_similarity_gt — a rule-based check that the response language matches the prompt language — and all scored 1.0. The remaining 7 cases mapped to a strategy named "auto" which does not exist in the strategy registry — another tool bug, not a model failure.

✓ Multi-turn context retention (English)

P327: A conversation started about corn cultivation correctly pivoted to rice when the user asked about soil pH — the model maintained the agricultural framing and gave agronomically accurate advice (pH 5.5–7.0 for rice). P328 similarly handled a livestock → cotton weed control jump without losing the farmer persona.

✓ Appropriate safety refusals

P36 and P326: Identical prompts about chemical lab equipment were correctly refused twice with context-sensitive refusals — not generic blocks. This is meaningful: the model understood the disability-assistance framing and declined specifically on harm grounds.

✓ Indian language generation quality

Bengali responses (P6848, P6842) were fluent and contextually appropriate. The pesticide response correctly identified specific plant types (aparajita, nayantara) mentioned by the user and gave region-appropriate advice. Assamese-mixed text (P5170) showed code-switching typical of real user queries.

△ Context drift under topic switches

P325: When asked about Hindi alphabets, the model responded with Punjabi script (ਅ, ਆ) — a cross-script confusion. This is a meaningful failure for educational deployments where script accuracy is non-negotiable.

✗ Evaluation scores: all null

The response analyzer must be run as a separate manual step after each test run. There is no automated post-run evaluation trigger. All Dialogue_Coherence scores in the report are empty because the LLM-as-judge step is decoupled from execution with no documentation on this workflow.

04 — Tool Critique

Bugs found and fixed

Five code-level bugs had to be patched before a single test run could complete. Each is documented as a GitHub issue. None of these are edge cases — they block the primary evaluation path described in the README.

  • #01

    Hardcoded port 8000 in TestCaseExecutorDashboard backend

    Line 769 of src/app/TestCaseExecutorDashboard/back-end/main.py creates the InterfaceManagerClient with base_url="http://localhost:8000" — ignoring the port configuration in config.json. The Interface Manager runs on 8001 by default, causing all test runs to fail with a 404.

    BLOCKING
  • #02

    Language filter passes string instead of list to SQLAlchemy IN clause

    get_testcases_by_metric() in src/lib/orm/DB.py calls Languages.lang_name.in_(lang_names) where lang_names is a raw string (e.g. "Tamil"). SQLAlchemy expects a list. Any test run with a language selected throws an ArgumentError.

    BLOCKING
  • #03

    Ollama OpenAI-compatible /v1 endpoint not supported in v0.24.0

    The _run_local() function constructs base_url + "/v1" and uses the OpenAI SDK. Ollama 0.24.0 returns 404 on /v1/chat/completions. The tool assumes a newer Ollama version without stating this requirement. Fixed by patching to use Ollama's native /api/chat endpoint directly.

    BLOCKING
  • #04

    File encoding error on Windows for Indian-language test data

    src/app/importer/main.py opens JSON files without specifying encoding, defaulting to Windows cp1252. DataPoints.json contains UTF-8 Indian language characters causing UnicodeDecodeError on Windows. All open() calls need encoding='utf-8'.

    WINDOWS-ONLY
  • #05

    SLOR metric incompatible with non-English responses

    The Fluency_Score strategy uses SLOR (Syntactic Log-Odds Ratio), an English-language metric. When the agent responds in Hindi, Bengali, or Assamese, SLOR throws a KeyError: 'SLOR' and returns score 0. There is no language-aware metric routing. For a tool targeting Indian AI systems, this is a fundamental design gap.

    DESIGN FLAW
05 — Conclusions

What can be concluded

On the model: Gemma3n:e2b demonstrates adequate conversational coherence for agricultural and healthcare queries in English and Bengali. Multi-turn context is maintained for domain-consistent conversations. The primary failure mode is cross-script confusion in educational contexts — a meaningful limitation for literacy applications.

On the tool: The CeRAI AIEvaluationTool is architecturally sound but not production-ready for the API + Indian language evaluation path. The five bugs above are not configuration issues — they require code changes. The tool works well for its original design target (WhatsApp/web bots in English) but the Indian language evaluation story is incomplete: metrics are English-only, the streaming/non-streaming distinction is undocumented, and the multi-service startup sequence has no orchestration.

06 — Limitations

Where findings should not be generalised

No quantitative scores

All findings are qualitative. The LLM-as-judge pipeline returned null scores for all Dialogue_Coherence cases. Conclusions about response quality are based on manual inspection of 10 cases — too small a sample for statistical claims.

Single model, single hardware configuration

Gemma3n:e2b on an RTX 3050 4GB is one configuration. Smaller models (llama3.2:1b) failed consistently on Indian language prompts. Model selection substantially changes results.

Test cases not verified for ground truth

The DataPoints.json corpus was used as-is. Expected responses were not reviewed for accuracy before the run. Some "correct" reference responses may themselves be wrong.

Windows-specific issues may not appear on Linux

The encoding bug (#04) and Node PATH issues are Windows-only. A Linux deployment may bypass several of the bugs documented here, making the tool appear more stable than it is cross-platform.
